# 代码分类与功能说明

本文档是日常开发时的工程归类手册。正式架构边界以 [ARCHITECTURE.md](ARCHITECTURE.md) 为准。

当前项目正式按五个主层分类，并固化 `P2.5 数据访问层` 作为 P1/P2 与持久化数据之间的独立边界：

1. `P0 安全与规则层`
2. `P1 协调层`
3. `P2 模块能力层`
4. `P2.5 数据访问层`
5. `P3 界面层`
6. `监控层`

## 一、层级速查表

| 层级 | 主要职责 | 优先查看位置 |
| --- | --- | --- |
| P0 安全与规则层 | 安全、脱敏、权限、治理、策略、确定性规则 | `mock-server/middleware`、`mock-server/services/*security*`、`mock-server/services/*Governance*`、`mock-server/plugins/*rule-engine*` |
| P1 协调层 | 任务分发、上下文装配、Assistant/Prompt/策略解析、Session 连续性、工作流编排 | `mock-server/routes/runtimeRoutes.js`、`mock-server/services/taskWorkbenchService.js`、`mock-server/services/sessionService.js`、`mock-server/services/pluginRegistryService.js` |
| P2 模块能力层 | 判断、检索、写作、模型、数据库、Python Runtime、外部数据源执行 | `mock-server/flows`、`mock-server/services/search*`、`mock-server/services/model*`、`mock-server/services/databaseService.js`、`python_runtime/app` |
| P2.5 数据访问层 | 外部连接 API Key 引用与历史会话/消息的内部持久化 | `mock-server/data`、`mock-server/data/routes`、`mock-server/data/models`、`mock-server/data/database.js` |
| P3 界面层 | 页面、组件、路由、前端接口、表单状态、结果展示 | `src/pages`、`src/components`、`src/api`、`src/utils`、`src/router`、`src/layout` |
| 监控层 | 健康检查、运行事件、Tracing、告警、回归脚本、验证报告 | `mock-server/services/opsObservabilityService.js`、`mock-server/tracing.js`、`scripts`、`mock-server/test-results` |

## 二、P0 安全与规则层

### 允许做

- 安全中间件、SSO、租户隔离、权限域校验。
- 脱敏、出站控制、Prompt 完整性校验、密钥托管。
- Assistant、Prompt、Settings、数据库关系等治理注册、版本、发布、回滚、审计。
- 规则引擎、检索规则、外发规则、策略白名单。
- 对不合规输入或执行链路进行拒绝、降级、fallback。

### 不允许做

- 不渲染页面。
- 不决定页面跳转。
- 不把表单默认文案写成规则来源。
- 不绕过 P1 直接完成业务链路。
- 不把监控告警直接当业务答案返回。

### 典型文件

- `mock-server/middleware/security.js`
- `mock-server/services/securityMiddlewareService.js`
- `mock-server/services/sanitizationService.js`
- `mock-server/services/secretVaultService.js`
- `mock-server/services/settingsGovernanceService.js`
- `mock-server/services/settingsGovernanceBridgeService.js`
- `mock-server/services/governanceRegistryService.js`
- `mock-server/services/governanceAuditService.js`
- `mock-server/plugins/rule-engine`
- `mock-server/plugins/search-rule-engine`
- `python_runtime/app/middleware/safety.py`

## 三、P1 协调层

### 允许做

- 接收运行态请求并组织执行上下文。
- 解析当前激活 Assistant、Prompt、策略、数据范围、数据库关系。
- 根据任务意图选择判断、检索、写作或工作台推荐。
- 创建和维护 Session、Step、Evidence、Continue Payload。
- 调度 P2 能力，并合并 P0 规则结果。
- 处理 workflow plugin、canary/stable、fallback、降级。

### 不允许做

- 不直接实现具体分析、检索、写作算法。
- 不持有页面 UI 状态。
- 不绕过 P0 安全检查。
- 不硬编码某个行业模板的页面默认文案。
- 不把模型、数据库、外部数据源底层细节暴露给界面层。

### 典型文件

- `mock-server/routes/runtimeRoutes.js`
- `mock-server/services/taskWorkbenchService.js`
- `mock-server/services/assistantContextService.js`
- `mock-server/services/sessionService.js`
- `mock-server/services/pluginRegistryService.js`
- `mock-server/services/workflowNodeRegistry.js`
- `mock-server/services/promptService.js`
- `mock-server/services/settingsService.js`

## 四、P2 模块能力层

### 允许做

- 实现判断、检索、写作三类业务能力。
- 调用模型、Python Runtime、数据库、检索连接器和外部数据源。
- 生成结构化业务结果，例如判断摘要、证据列表、参考文稿。
- 根据 P1 传入的上下文和 P0 规则约束执行能力。
- 提供模块内 fallback，并把 fallback 结果交回 P1 装配。

### 不允许做

- 不直接读写页面状态。
- 不决定当前激活 Assistant 或治理版本。
- 不跳过 P1 自行创建完整会话链路。
- 不把安全、权限、出站控制当作普通业务逻辑随意处理。
- 不把监控日志当作生产配置或业务事实来源。

## 五、P2.5 数据访问层

### 允许做

- 提供会话、消息、外部连接密钥的内部 REST 接口。
- 使用 SQLite 做轻量持久化，并保留后续替换数据库的边界。
- 为 P1 协调层提供历史会话上下文和对话消息。
- 保存外部连接 API Key 引用和 `has_api_key` 状态，真实密钥只放在进程内环境变量或后续密钥仓库中。
- 提供初始化脚本，只初始化内部数据库结构。
- SQLite 写入失败时允许 P1 降级回原有 Redis/内存/文件存储。

### 不允许做

- 不执行判断、检索、写作等业务能力。
- 不保存或决定当前启用模型、本地大模型、云模型连接、模型路由策略。
- 不保存或决定 Assistant 激活、Prompt 发布、Prompt 模板、工作流策略或页面跳转。
- 不绕过 P0 暴露敏感数据。
- 不直接处理前端状态。
- 不作为前端公开 API 使用，只允许后端内部直接调用或通过 `/internal/data/*` 本地内部路由访问。

### 典型文件

- `mock-server/data/database.js`
- `mock-server/data/models/session.js`
- `mock-server/data/models/externalConnection.js`
- `mock-server/data/routes/internalData.js`
- `mock-server/data/seed.js`

运行态数据库、密钥仓库和调用日志只保留在本地，例如 `mock-server/data/*.db`、`data/*.db`、`data/secretVault.json`、`*.jsonl`，不进入 Git 提交。

## 六、P3 界面层

### 允许做

- 页面展示、表单交互、按钮流转、结果卡片、错误提示。
- 通过 `src/api` 调用后端接口。
- 做轻量前端适配，例如页面表单默认值、Session 恢复、导航状态承接。
- 展示后端返回的 Assistant、Prompt、模型、数据库、Evidence、运行上下文和告警摘要。
- 把用户操作转换成明确的接口 payload。

### 不允许做

- 不实现核心判断、检索、写作规则。
- 不直接读写后端数据文件、数据库或治理注册表。
- 不把行业模板默认值硬编码成真实来源。
- 不绕过 `src/api` 拼接后端细节。
- 不把页面状态当成 Session 或治理事实来源。

### 典型文件

- `src/pages/Search`

## 七、监控层

### 允许做

- 记录服务启动、请求量、成功率、失败率、耗时、token、成本、fallback。
- 记录 Python Runtime、mock-server、Vite、Jaeger 等健康状态。
- 生成运行看板、告警、回归测试报告。
- 为上线前检查提供验证证据。

### 不允许做

- 不生成业务判断、检索证据或写作内容。
- 不绕过 P0/P1/P2 修改业务结果。
- 不自动删除、屏蔽或重写用户数据，除非 P0 明确授权。
- 不把测试结果文件当作生产配置。
- 不静默吞掉业务错误。

### 典型文件

- `mock-server/services/opsObservabilityService.js`
- `mock-server/tracing.js`
- `mock-server/routes/settingsRoutes.js` 中的运维与健康接口
- `scripts/local-stack.mjs`
- `scripts/settings-governance-regression.mjs`
- `scripts/workflow-release-regression.mjs`
- `scripts/workflow-platform-regression.mjs`
- `mock-server/test-results`
- `data/opsRuntimeDashboard.json`

`mock-server/test-results`、`test-results`、`test-evidence` 和 `data/opsRuntimeDashboard.json` 是本地验证或运行态观测产物，默认不进入 Git 提交。

## 九、开发边界约定

- 页面层只负责展示和交互，不直接写复杂业务规则。
- 前端可以读取当前 Assistant 默认值，但默认值来源必须来自后端治理数据。
- 接口层只负责请求、响应转换和轻量适配，不承担核心业务判断。
- 协调层负责流程组织，不承担具体模块算法。
- 模块层负责能力执行，不承担全局治理和 UI 决策。
- 数据访问层负责结构化数据读写，不承担业务能力和治理决策。
- 本地大模型、云模型连接和模型路由必须留在 P2 模块插件，不进入 P2.5。
- 安全与规则层的拒绝、降级、审计必须结构化返回。
- 监控层默认只读观察，不改变业务结果。

## 十、落地结论

后续开发时，如果不确定文件应该放哪，按下面顺序判断：

1. 是否涉及安全、权限、脱敏、治理、规则？是则进入 P0。
2. 是否涉及任务分发、上下文、Session、Assistant/Prompt/策略解析？是则进入 P1。
3. 是否涉及判断、检索、写作、模型、数据库、Python Runtime 能力执行？是则进入 P2。
4. 是否涉及历史会话、消息、外部连接 API Key 的内部结构化读写接口？是则进入 P2.5。
5. 是否涉及页面、组件、前端请求、用户交互？是则进入 P3。
6. 是否涉及健康、日志、Tracing、告警、回归验证？是则进入监控层。

如果一个需求同时跨多层，必须拆开实现，不允许把规则、流程、能力、界面和监控混在同一个文件里。
