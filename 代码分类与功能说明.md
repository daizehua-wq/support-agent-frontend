# 代码分类与功能说明

本文档用于帮助后续开发时快速判断：

- 某个文件属于哪一类代码
- 某个功能应该优先去哪个目录找
- 新增文件时应该放在哪一层

当前项目按职责分为三类：

1. 前端页面代码
2. 接口层
3. 服务端逻辑

## 一、分类规则

### 1. 前端页面代码

负责页面展示、交互、路由、组件编排，以及小程序页面入口。

归属目录：

- `src/pages`
- `src/components`
- `src/layout`
- `src/router`
- `src/App.tsx`
- `src/main.tsx`
- `miniprogram/pages`
- `miniprogram/app.js`
- `miniprogram/app.json`
- `miniprogram/app.wxss`

适合放进这一类的内容：

- 页面 UI
- 页面状态管理
- 页面表单交互
- 页面跳转
- 页面内展示组件
- 页面摘要卡片

### 2. 接口层

负责前端和后端之间的数据连接，包括请求封装、接口调用、错误处理、会话恢复辅助。

归属目录：

- `src/api`
- `src/utils`

适合放进这一类的内容：

- HTTP 请求封装
- 接口方法定义
- 响应结构处理
- 接口错误转换
- 从历史会话中恢复页面上下文的工具函数

### 3. 服务端逻辑

负责业务执行、运行链路、治理逻辑、配置读写、模型路由、数据库能力、Python Runtime、工程脚本。

归属目录：

- `mock-server`
- `python_runtime`
- `config`
- `runtime`
- `scripts`
- `vite.config.ts`
- `eslint.config.js`
- `index.html`

适合放进这一类的内容：

- Express 路由
- 业务服务
- 数据治理逻辑
- 模型调用与路由
- 配置治理
- 回归脚本
- 启动脚本

## 二、三类代码分别负责什么

| 分类 | 主要职责 | 开发时优先查看的位置 |
| --- | --- | --- |
| 前端页面代码 | 展示页面、收集输入、展示结果、组织交互流程 | `src/pages`、`src/components` |
| 接口层 | 调用后端接口、处理响应、统一错误、恢复历史会话上下文 | `src/api`、`src/utils` |
| 服务端逻辑 | 承接接口请求、执行业务逻辑、治理配置、模型/数据库/Python 运行时联动 | `mock-server`、`python_runtime`、`scripts` |

## 三、前端页面代码说明

### 1. 应用骨架

- `src/main.tsx`
  负责挂载 React 应用。
- `src/App.tsx`
  负责整体应用入口。
- `src/router/index.tsx`
  负责路由分发，把页面挂到不同路径。
- `src/layout/MainLayout.tsx`
  负责顶部导航和整体页面布局。

### 2. 页面模块

#### `src/pages/Home`

首页入口，负责：

- 展示平台概览
- 展示当前激活模板
- 展示最近会话
- 提供各模块快捷入口

#### `src/pages/Workbench`

统一任务工作台，负责：

- 接收用户自然语言任务
- 识别任务意图
- 选择助手模板
- 生成任务材料包
- 推荐后续流转到判断、检索或写作页面

#### `src/pages/Analyze`

任务判断页，负责：

- 把任务拆成目标、风险、建议
- 展示判断结果
- 展示执行上下文
- 展示 Prompt、模型、数据库摘要
- 支持从历史会话恢复

#### `src/pages/Search`

资料检索页，负责：

- 根据主题检索资料
- 展示证据列表
- 展示证据级别和外发状态
- 汇总检索结果
- 支持从历史会话续接

#### `src/pages/Script`

参考写作页，负责：

- 根据任务目标和上下文生成参考文稿
- 结合历史分析结果和检索证据继续写作
- 输出邮件、说明、方案、纪要等内容

#### `src/pages/AssistantCenter`

助手模板和 Prompt 治理台，负责：

- 管理 Assistant 模板
- 管理 Prompt
- 管理模板作用域和默认策略
- 处理模板发布、激活、历史记录

#### `src/pages/ModelCenter`

模型治理台，负责：

- 管理模型资源
- 设置默认模型
- 管理模块绑定
- 管理 fallback 策略
- 测试模型连接和结果

#### `src/pages/DatabaseManager`

数据库治理台，负责：

- 管理数据库列表和详情
- 健康检查
- 轻绑定关系维护
- 管理外部数据源
- 运行外部数据源 query、fetch、download

#### `src/pages/Settings`

系统设置页，负责：

- 系统基础设置
- 运行时状态总览
- Python Runtime 配置
- 工作流发布配置
- 治理发布与回滚
- 安全态势和运维看板

### 3. 通用组件

#### `src/components/common`

通用展示组件，负责：

- 页面标题
- 通用结果卡片
- 加载态
- 空状态
- 复制按钮

#### `src/components/card`

摘要卡片组件，负责：

- 绑定关系摘要
- 数据库关系摘要
- 执行结果摘要
- 版本信息展示

#### `src/components/governance`

治理展示组件，负责：

- 版本历史
- 变更记录展示

### 4. 小程序

#### `miniprogram`

当前主要是轻量入口和测试页，职责较简单：

- `miniprogram/pages/home`
  小程序首页入口
- `miniprogram/pages/test`
  小程序测试页面
- `miniprogram/app.js`
  小程序全局入口

## 七、新文件归类建议

后续新增文件时，建议按下面规则放置：

- 新页面、新页面局部组件：放到 `src/pages/<模块名>` 或 `src/components`
- 新接口方法：放到 `src/api`
- 新的前端错误处理或会话恢复工具：放到 `src/utils`
- 新业务路由：放到 `mock-server/routes`
- 新业务服务：放到 `mock-server/services`
- 新流程编排：放到 `mock-server/flows`
- 新 Python 接口或 Python 能力：放到 `python_runtime/app`
- 新回归脚本或启动脚本：放到 `scripts`

## 八、边界约定

为避免职责混乱，后续开发建议保持下面约定：

- 页面层只负责展示和交互，不直接写复杂业务规则
- 接口层只负责请求和数据转换，不承担核心业务判断
- 服务端逻辑负责真正的规则、治理、执行和落库
- 配置、治理、模型路由优先收口到 `mock-server/services` 和 `mock-server/routes`
- 若前端页面需要复杂摘要，优先由后端返回结构化结果，前端负责展示

## 九、当前结论

这个项目的整体结构可以概括为：

- 前端页面代码：负责工作台、治理台、结果展示
- 接口层：负责前后端连接和会话恢复
- 服务端逻辑：负责运行、治理、模型、数据库、脚本和 Python Runtime

后续开发时如果不确定文件应该放哪，优先先判断它属于：

- 展示和交互
- 数据连接
- 业务执行

然后再决定放到前端页面代码、接口层或服务端逻辑。
